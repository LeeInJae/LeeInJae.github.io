EMC_35. 스레드 기반 프로그래밍보다 과제 기반 프로그래밍을 선호하라

doAsyncWork라는 함수를 비동기적으로 실행한다고 하자. 방법은 크게 2가지가 있다.

  • std::thread 객체를 생성해서 그 객체에서 doAsyncWork를 실행(thread 기반)
  • 또 하나는 doAsyncWork를 std::async에 넘겨서 실행(task 기반)

대체로 task 기반이 thared 기반보다 우월하다.

반환값의 차이

task 기반에서는 반환값을 돌려주는데, thread기반에서는 반환값을 돌려주지 않는다. 대부분 코드에서 doAsyncWork가 돌려주는 반환값에 관심이 있을 것이다라는 가정은 옳다. task 기반에서는 std::async가 돌려주는 미래 객체에 get이라는 멤버함수를 통해서 반환값을 받아올 수 있다.혹시나 doAsyncWork가 예외를 뱉어냈다면 thread 기반에서는 std::terminate를 호출해서 바로 죽지만, task 기반에서는 get을 통해 그 예외에도 접근할 수 있다.

스레드의 의미

  • 실제 계산을 수행하는 스레드를 뜻하는 하드웨어 스레드(코어 하나당 한개의 스레드)
  • 운영체제가 하드웨어 스레드들에서 실행되는 모든 프로세서와 일정을 관리하는데 사용하는 os 스레드(시스템 스레드라고도 함)
  • c++표준 라이브러리 std::thread. 하나의 C++ 프로세스 안에서 std::thread 객체는 바탕 소프트웨어 스레드에 대한 핸들로 작용

소프트웨어적 스레드는 제한된 자원이고 한계점을 초과한다면 std::system_error에외가 발생한다.

과다구독

과다구독이란 실행 준비가 된(차단되지 않은) 소프트웨어 스레드가 하드웨어 스레드보다 많은 상황을 말한다. 이렇게 되면 스레드 스케쥴러에 의해 하드웨어 실행 시간을 여러조각으로 나누어 소프트웨어 스레드에게 나누게된다. 이렇게 나뉜 시간을 time slice라고 하면 소프트웨어 스레드가 교체될때 context switch가 수행되고 이것은 전반적인 시스템 스레드 관리 부담의 증가로 이어진다. 이는 CPU 캐쉬또한 망치게해서 성능의 하락을 야기한다.

  • 새로운 소프트웨어 스레드를 실행하면 그 코어에서 실행되던 그리고 다음번에도 그 코어에서 실행될 가능성이 큰 기존 스레드들에 대한 캐쉬가 바뀌기 때문이다

과다구독은 피하기 어렵고, CPU 아키텍쳐, CPU캐쉬 친화적인 정도 등등에 따라 해법이 달라질 수 있다. 따라서 이쪽에서 잘돌아가더라도 저쪽에서 잘 안돌아갈 여지가 있다는 것이다. 이런 문제들을 누군가에게 떠넘긴다면 좋아질 것 같은데 바로 std::async를 사용하면 된다.

이 호출은 스레드 관리의 책임을 독자에서 C++ 표준 라이브러리 구현자로 옮긴다. 이 호출은 std::thread와 달리 새 소프트웨어 스레드를 생성하지 않을수도 있다. 대신 std::async는 지정된 함수를 doAsyncWork의 결과가 필요한 스레드(즉, fut에대해 get이나 wait를 호출하는 thread)에서 실행하라고 스케쥴러에게 요청할 수 있으며, 합리적인 스케쥴러는 시스템이 과다구독되었거나 스레드가 부족한 상황에서 그러한 자유의 장점을 취한다. 이 호출이 부하 불균형 문제를 워넌적으로 막지는 못할 수 있다. 하지만 load balancing에 관련해서 프로그래머보다 스케쥴러가 훨씬 더 많은 정보들을 가지고 있기때문에 훨씬 유리하다.

스레드를 일일이 관리해야하는 수고로움이 없다

과제 기반 설계는 비동기적으로 실행된 함수의 결과를 자연스럽게 조회할 수 있고, 스레드에 대한 관리를 일일이 해야하는 수고도 덜 수 있따. 하지만 스레드를 직접 다루는게 적합한 경우도 존재한다. 다음의 예를 보자.

  • 바탕 스레드 적용 라이브러리의 API에 접근해야하는 경우. 바탕 스레드 적용 라이브러리의 API에 접근 할 수 있도록, std::thread 객체는 흔히 native_handle이라는 멤버 함수를 제공한다. 그런데 std::future(std::async가 돌려주는 객체)에는 이애 해당하는 기능이 없다. c++ 에서 제공하는 동시성 API 수준보다 저수준 플랫폼 고유 API(pthreads라이브러리, Windows 스레드 라이브러리) 등에 접근하고자 할때는 std::thread가 필요하다
  • 응용 프로그램의 스레드 사용량을 최적화해야 하는, 그리고 할 수 있어야 하는 경우. 미리 정해진 컴퓨터에서 유일하게 의미있는 프로세스로 실행될 서버 소프트웨어를 개발하는 경우가 이 예이다.
  • c++ 동시성 API가 제공하는 것 이상의 스레드 적용 기술을 구현해야 하는 경우. 이를테면 C++구현이 스레드 풀을 제공하지 않는 특정 플랫폼을 위해 스레드 풀을 직접 구현해야하는 경우가 이 예이다.

이러한 예외들은 흔치않으면 thread 기반 보다 task 기반 설계가 바람직하다.

정리

  • std::thread API에서는 비동기적으로 실행된 함수의 반환값을 직접 얻을수 없으며, 만일 그런 함수가 예외를 던지면 프로그램이 종료된다.
  • 스레드 기반 프로그래밍에서는 스레드 고갈, 과다구독, 부하 균형화, 새 플랫폼으로의 적응을 독자가 직접 처리
  • std::async와 기본 시동 방침을 이용한 과제 기반 프로그래밍은 그런 대부분의 문제를 알아서 처리.
